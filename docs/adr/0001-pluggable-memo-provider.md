# ADR-0001: Pluggable memo provider with deterministic + LLM modes

**Status:** Accepted
**Date:** 2026-05-09
**Authors:** Aakash Vidiyala

## Context

The underwriting packet has one piece that is fundamentally non-deterministic: the **memo** — the prose synthesis a reviewer reads alongside the structured findings. Everything else in a packet (risk signal, action plan, claims timeline, citations, snapshot hash, rubric version) is computed from rules and is reproducible by construction. The memo is the LLM's job.

This created a tension we had to resolve before scaling either side of the product:

1. **Live mode needs a real LLM.** Carrier-facing memos read better when written by a frontier model than by a template. We need to use Anthropic in production.
2. **Eval mode needs reproducibility.** The eval scoreboard at `/evals` runs 15 scenarios across 7 exposure classes and grades them against deterministic scorers. If the memo varies between runs, we can't measure quality drift, can't bisect regressions, and can't publish a stable scoreboard.
3. **The audit trail needs to know which path was taken.** A packet generated via deterministic fallback (because the Anthropic call failed) should be visibly distinguishable from one generated by Claude — both for reviewer trust and for incident postmortems.

A naïve solution — call the LLM in production and skip it in tests — solves (1) and (2) but breaks (3): the same code path produces different artifacts depending on environment, and the artifact itself doesn't record which path ran.

## Decision

Define a `MemoProvider` ABC in `backend/app/providers/base.py` with two production implementations:

- **`DeterministicProvider`** — template-based memo drafting with no external dependencies. Produces consistent, explainable output driven entirely by the structured packet findings. Used by the eval harness in snapshot mode and as the fallback when the LLM call fails.
- **`AnthropicProvider`** — LLM-assisted memo drafting via Claude (`claude-haiku-4-5-20251001`). Requires `ANTHROPIC_API_KEY`. Raises `ProviderNotConfiguredError` if the key is missing; the caller is expected to fall back to `DeterministicProvider`.

Both providers return a `MemoOutput` dataclass that carries `provider_name`, `mode` (`deterministic` | `llm`), and `model`. The `UnderwritingMemo` schema persists these on the packet alongside an optional `fallback_reason` field that is populated when an LLM call failed and the deterministic path was used as a fallback.

The interface is intentionally **bounded and auditable**:

- Providers receive only structured, validated findings (risk type, severity, citation excerpts) — never raw user input. This eliminates a large class of prompt-injection and PII-leakage concerns.
- Provider output is stored as draft text, never as a compliance decision. The rubric engine and citation validator run before and independently of any provider call.
- Provenance (`provider`, `model`, `fallback_reason`) is part of the packet's persistent record, not just a log line.

## Consequences

**Positive**

- **Reproducible evals.** The scoreboard runs `DeterministicProvider` by default, producing a stable JSON snapshot that the public `/evals` dashboard reads. Live-mode runs (under `--provider anthropic`) write a fresh snapshot tagged with `ProviderInfo`, so we can compare drift between deterministic and LLM output without conflating the two.
- **Honest fallbacks.** When Anthropic is rate-limited or returns an error, the packet still ships — but `fallback_reason` records why, and `provider` reflects the deterministic path. Reviewers can filter for "real LLM packets" vs "fallback packets" without code archaeology.
- **Provider swap is one line.** Swapping in a Gemini, Bedrock, or local-Ollama backend is implementing the ABC and adding a constructor branch. The agent runtime, the eval harness, and the packet schema don't move.
- **Prompt caching applies cleanly.** The `SYSTEM_PROMPT` block on `AnthropicProvider` is marked with `cache_control: ephemeral` (commit `ee93716`), so back-to-back evals or busy nights amortize the system tokens across calls. This works because the system block is genuinely stable — a property the deterministic-vs-LLM split surfaced and forced us to preserve.

**Negative**

- **Two memo styles to maintain.** When we add a new risk type or severity tier, both providers need updating. Mitigation: the deterministic templates are keyed off the same `risk_type` / `severity` enums the LLM prompt references, so divergence is visible in code review.
- **The deterministic memos read like templates.** They are templates. For a real underwriter, the LLM memo is meaningfully better; the deterministic version is functionally adequate but visibly machine-written. We accept this because the deterministic memo's job is to be *consistent and reviewable*, not to delight.
- **The `fallback_reason` field is one more thing reviewers need to learn to read.** Documented in `CONTEXT.md` and surfaced (eventually) in the packet UI via a small "Generated by …" badge.

## Alternatives considered

- **LLM-only with snapshot caching.** Run Claude in production and cache responses keyed on packet hash for evals. Rejected: cache invalidation becomes the dominant operational concern, and "deterministic mode" in eval becomes "deterministic *given the right cache state*" — fragile.
- **Deterministic-only.** Skip the LLM entirely and generate templated memos always. Rejected: the memo is the most reviewer-facing artifact in the system; templating it caps the product's perceived quality at "ML-adjacent SaaS dashboard" instead of "AI-native underwriting tool."
- **Conditional LLM call inside one provider class.** A single class with `if env == "test": template else: llm`. Rejected: hides the dependency boundary, makes the LLM/non-LLM contract implicit, and provides no clean place to record provenance.
